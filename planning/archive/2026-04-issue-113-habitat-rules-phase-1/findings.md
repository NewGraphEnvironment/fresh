# Findings: Issue #113 — Habitat eligibility rules format Phase 1

## The problem

`parameters_habitat_thresholds.csv` (fresh 0.11.0) handles single-rule species cleanly via the `spawn_edge_types` / `rear_edge_types` columns. But several species in bcfishpass v0.5.0 need MULTIPLE rules joined by OR:

- **SK rear**: lake-only (`waterbody_type=L AND area_ha >= 200`), no stream rearing
- **CO rear**: stream/canal with thresholds OR wetland-flow (`edge_type IN (1050, 1150)`) with NO threshold checks
- **All anadromous spawn**: stream/canal OR river polygon (`waterbody_type=R`)
- **BT/CO/CH/ST/WCT/RB rear**: stream/canal OR river polygon OR wetland-flow OR lake — four rules

The CSV's `*_edge_types` column can only express ONE rule. Multi-rule species can't fit.

## Why a YAML format

Each rule has up to 6 optional predicates (`edge_types`, `edge_types_explicit`, `waterbody_type`, `lake_ha_min`, `gradient`, `channel_width`). Squeezing this into a CSV would require:
- One row per rule (proliferates rows, hard to read)
- Or multiple columns per rule (hard to scale, lots of NAs)

YAML naturally expresses lists of records with optional fields. `yaml` package is small, well-maintained, pure R, supports comments.

## Critical design choice: rules YAML semantics vs CSV semantics

**Two layers**:
1. **CSV** = thresholds (gradient, channel_width — the numeric ranges)
2. **YAML rules** = WHICH segments qualify (which edge types, which waterbody types, etc.)

Rules `thresholds: true` (default) inherits the CSV thresholds and ANDs them with the rule's predicates. Rules `thresholds: false` skips inheritance — the rule stands alone (this is the wetland-flow carve-out: ANY 1050/1150 segment qualifies as CO rearing regardless of gradient or width).

## Sync from link

The bundled YAML is generated by `link/inst/extdata/parameters_habitat_dimensions.csv` — link contains the biological dimensions / rationale per species. fresh treats the rules YAML as a derived artifact:

```bash
# In the fresh repo:
cp ../link/inst/extdata/parameters_habitat_rules.yaml inst/extdata/
```

This is documented in the YAML header. Don't hand-edit the YAML in fresh — edit the dimensions CSV in link and regenerate.

## Lake rearing column

The existing `lake_rearing` boolean column logic stays unchanged. It derives from CSV `rear_channel_width_min/max` + lake polygon join. The rules YAML controls the `rearing` column. This separation:
- Reduces blast radius (no concurrent change to two columns)
- Keeps `lake_rearing` available for consumers who want it independently
- Lets SK's lake-only rule populate `rearing` (not `lake_rearing`) — which is what bcfishpass does

## Backward compat strategy

Two paths:

1. **Default**: `frs_habitat()` calls `frs_params()` which loads the bundled YAML. Every species gets rules. Behavior changes for SK (rear=0 on streams), PK/CM (rear=FALSE everywhere), and others as the rules dictate.

2. **Opt out**: `frs_params(rules_yaml = NULL)` explicitly skips rules. No species get `$rules` attached. The classify loop falls through to the existing `$ranges` path. This preserves the v0.11.0 behavior verbatim.

The v0.11.0 → v0.12.0 release will document the default change in NEWS.md and explain how to opt out for users who want pre-rules behavior.

## Phase 1 vs Phase 2 (#114)

Phase 1 excludes the `mad` predicate because it requires enriching segmented streams with `mad_m3s` from `whse_basemapping.fwa_stream_networks_discharge`. That's a separate piece of work tracked in #114.

If a rules YAML contains `mad`, the parser errors with a clear message pointing to #114.

## Test plan

Use ADMS sub-basin (`wscode_ltree <@ '100.190442.999098.995997.058910.432966'::ltree`) — 188 base segments.

Functional tests of the rules format:
- SK rear stream count → 0 (currently 21.55 km on streams pre-rules)
- CO rear includes 1050/1150 segments regardless of gradient/cw
- BT spawn includes river-polygon segments
- `frs_params(rules_yaml = NULL)` matches v0.11.0 behavior

NOT tested in fresh:
- bcfishpass-exact match counts (that's link's job)
- bundled YAML correctness (that's link's responsibility, derived from biological dimensions)

## Risks

- **Default behavior change**: SK rear km drops from 21.55 to 0 on ADMS. PK/CM lose rearing entirely. This is intentional but must be loud in NEWS.
- **YAML parsing brittleness**: silent typos (e.g., `waterbody_type: l` lowercase) need explicit validation.
- **Predicate explosion**: future rules might want NOT, nested OR/AND. Phase 1 keeps it flat (AND inside rule, OR across rules). Document the limit.
